# v1.29.3 - Penguatan Login dan Audit Autentikasi

## Ringkasan

Rilis patch ini menggabungkan perbaikan issue #224 dan #225. Fokusnya adalah memperkuat keandalan audit autentikasi dan memastikan halaman login kembali mengikuti sumber kebenaran form Django melalui crispy-forms sebelum pekerjaan penyederhanaan UI dilanjutkan.

## Dampak Keamanan

Sebelumnya, log autentikasi menerima nilai pertama dari header `X-Forwarded-For` setiap kali header tersebut ada. Request langsung ke Django dapat mengirim header palsu dan membuat event `login_success` atau `login_failed` mencatat IP yang dikendalikan attacker.

Mulai rilis ini, log autentikasi dan log error terpusat menggunakan `REMOTE_ADDR` secara default. Header `X-Forwarded-For` hanya dipakai bila request datang dari reverse proxy langsung yang secara eksplisit tercantum di `AUTH_AUDIT_TRUSTED_PROXIES`, dan chain IP yang diterima harus valid secara sintaks.

## Perubahan Utama

- Menambahkan resolver `apps.core.client_ip.get_client_ip()` untuk derivasi IP client yang aman bagi log keamanan.
- Menggunakan resolver tersebut pada event `login_success`, `login_failed`, `logout`, serta log error terpusat.
- Menambahkan konfigurasi environment `AUTH_AUDIT_TRUSTED_PROXIES` untuk deployment yang berada di belakang reverse proxy tepercaya.
- Mengubah `/login/` agar memakai Django `LoginView` dengan `apps.core.forms.CrispyAuthenticationForm`.
- Mengganti input username/password manual di `registration/login.html` dengan render `{% crispy form %}` dari bound `AuthenticationForm`.
- Menghapus JavaScript password-toggle dan CSS lama yang hanya dipakai oleh markup input manual yang sudah dihapus.

## Perilaku Setelah Upgrade

Tidak ada migration database.

Untuk deployment tanpa reverse proxy tepercaya, biarkan `AUTH_AUDIT_TRUSTED_PROXIES` kosong. Dalam mode ini, log autentikasi mengabaikan `X-Forwarded-For` dari client dan memakai `REMOTE_ADDR`.

Untuk deployment di belakang reverse proxy, isi hanya dengan IP atau CIDR proxy langsung yang Anda kontrol. Proxy tersebut wajib menghapus header `X-Forwarded-For` dari client sebelum menulis nilai canonical.

Contoh:

```env
AUTH_AUDIT_TRUSTED_PROXIES=10.10.0.1,10.10.0.0/24
```

## Catatan Login Form

Validasi kredensial tetap dijalankan oleh Django `AuthenticationForm`, perlindungan brute-force tetap melalui django-axes, dan redirect setelah login tetap divalidasi oleh `LoginView`. Nilai `next` internal yang aman tetap diarahkan sesuai target, sedangkan `next` eksternal yang tidak aman akan jatuh kembali ke redirect dashboard.
