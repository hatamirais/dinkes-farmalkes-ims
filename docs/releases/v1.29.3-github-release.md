# v1.29.3 - Authentication Audit IP Hardening

## Summary

This release fixes issue #224 by hardening how authentication and centralized error logs derive the client IP address. Audit logs now use `REMOTE_ADDR` by default and accept `X-Forwarded-For` only when the immediate upstream peer is explicitly configured as a trusted reverse proxy.

## Security Impact

Before this change, login success and login failure signal logs trusted the first `X-Forwarded-For` value whenever the header was present. A direct client could send a forged header and cause authentication audit logs to store attacker-controlled source IP data.

That weakened incident response and account-abuse investigation because operators could not rely on logged IPs for login activity when requests bypassed or reached Django directly.

## What Changed

- Added `apps.core.client_ip.get_client_ip()` as the single trusted resolver for security-relevant request IP logging.
- Updated authentication audit signals to use the shared resolver for `login_success`, `login_failed`, and `logout` events.
- Updated centralized error logging to use the same resolver, removing the duplicate unsafe pattern from `apps.core.views`.
- Added `AUTH_AUDIT_TRUSTED_PROXIES` for deployment-controlled proxy IPs/CIDRs.
- Documented that `X-Forwarded-For` is accepted only from a trusted immediate proxy that strips client-supplied forwarded headers.

## Deployment Notes

No database migration is required.

By default, leave `AUTH_AUDIT_TRUSTED_PROXIES` empty. In that mode, authentication audit logs derive client IPs from `REMOTE_ADDR` and ignore client-supplied `X-Forwarded-For`.

Set `AUTH_AUDIT_TRUSTED_PROXIES` only when Django receives requests from a reverse proxy you control. The proxy must remove inbound `X-Forwarded-For` headers from clients and set the canonical forwarded chain itself.

Example:

```env
AUTH_AUDIT_TRUSTED_PROXIES=10.10.0.1,10.10.0.0/24
```
