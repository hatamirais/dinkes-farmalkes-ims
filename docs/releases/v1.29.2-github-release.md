# v1.29.2

Patch release `1.29.2` hardens the public login surface before the planned UI simplification work. The release focuses on brute-force resilience, reducing unauthenticated operational disclosure, and documenting the active lockout semantics across the project references.

## Security

- Login lockout now uses django-axes username-based semantics with `AXES_LOCKOUT_PARAMETERS = ["username"]`.
- Repeated failures against the same username are now blocked even when attempts rotate through different source IP addresses.
- Public login and lockout messages no longer disclose the exact failed-attempt threshold.

## Root Cause

The previous configuration used a combined username/IP tuple: `AXES_LOCKOUT_PARAMETERS = [["username", "ip_address"]]`. That protected against repeated failures from the same source against the same username, but it gave every new source IP a separate counter for the target username. For predictable internal staff usernames, that left the login page more exposed to distributed password spraying than intended. The fix intentionally avoids a standalone source-IP lockout because, without trusted proxy-aware client IP extraction, an IP-wide counter can collapse all users behind one reverse proxy or load balancer into the same lockout bucket.

## User And Operator Impact

- Legitimate users see the same login flow and the same temporary lockout experience.
- Operators still receive a clear lockout message, but unauthenticated users no longer see the precise threshold value in public copy.
- Administrators should expect lockout counters to be stricter for repeated failures against a single username across multiple IPs.
- No database migration, Redis, Celery, cache broker, or infrastructure change is required.

## Technical Notes

- The change stays within the existing django-axes integration and does not introduce a second anonymous-login throttling system.
- Standalone IP-wide lockout remains deferred until proxy-aware client IP handling is configured and tested.
- The app remains compatible with the repository constraint of using local/in-process cache only.
- Project documentation was updated in `AGENTS.md`, `README.md`, and `SYSTEM_MODEL.md` so the security model matches the implemented setting.
